  1. 목적/범위
  - 이 dummy 데이터의 최종 용도가 "PaddleOCR 인식 성능 테스트용"인지, "모델
  학습(fine-tuning)용 데이터셋"인지 — 후자면 ground truth 라벨(정답 CSV)도 같이
  생성해서 나중에 OCR 결과와 비교(정확도 측정)할 수 있어야 함.
  - 생성 수량 (예: 100개, 1000개 등)

  2. 통장사본에 들어갈 필드 스키마
  - 은행명 (실제 은행명/로고를 쓸지, 가상 은행명을 쓸지 — 실제 은행 브랜드/로고
  사용은 저작권·상표 이슈가 있어 가상 은행명 추천)
  - 계좌번호 (은행별 자릿수/하이픈 포맷 규칙)
  - 예금주명 (한글 이름 랜덤 생성 규칙 — 성씨 리스트, 이름 조합 등)
  - 지점명, 계좌 개설일, 계좌 종류(입출금/예금/적금 등)
  - 잔액/거래내역 표 포함 여부 (거래일자, 적요, 입금액, 출금액, 잔액 컬럼 등)
  - 실제 통장사본에 흔히 있는 도장/서명 이미지, 바코드, 워터마크 등 포함 여부

  3. 랜덤화 규칙
  - 각 필드별 값의 범위/분포 (예: 잔액 0~1억, 날짜 범위 최근 5년 등)
  - 계좌번호/이름 등이 매번 유일해야 하는지(중복 방지)

  4. 레이아웃/디자인
  - 실제 통장 사본처럼 보이게 할 스타일 (표 기반 단순 레이아웃인지, 은행별
  템플릿 여러 종류를 흉내낼지)
  - 한글 폰트 지정 필요 (md→pdf 변환 시 한글 깨짐 방지용 폰트 파일 확보 필요)
  - 스캔한 것처럼 보이게 노이즈/회전/blur 등 augmentation을 넣을지 (실제 스캔
  문서 OCR 테스트라면 이런 왜곡이 오히려 중요한 테스트 요소)

  5. 기술 파이프라인
  - .md → PDF 변환 도구 선택 (pandoc+wkhtmltopdf, pandoc+xelatex, weasyprint 등
  — 한글 지원 여부 확인 필요)
  - PDF → PNG 변환 방법 (pdf2image/poppler 등), 해상도(DPI) — OCR 인식률에 직접
  영향
  - 출력 파일 네이밍 규칙, 저장 경로 구조

  6. Ground truth 저장 형식
  - OCR 결과 CSV와 비교하려면 생성 시점의 실제 값(정답)도 별도 CSV/JSON으로
  저장해야 함 — 컬럼 구성을 미리 정의

  이 중에서 특히 (1) 실제 은행명 사용 여부, (2) ground truth 라벨링이 필요한지,
  (3) 스캔 왜곡(노이즈) 포함 여부 세 가지가 코드 구조를 크게 좌우하니 먼저 답을
  주시면 좋을 것 같아요.

● 환경 확인해보니 pandoc CLI, paddleocr/paddlepaddle/pillow는 이미 설치돼
  있어요. 파이프라인(.md → HTML → PDF → PNG → 왜곡)을 Windows에서 외부
  바이너리(wkhtmltopdf, poppler, LaTeX) 설치 없이 pip만으로 돌아가게 구성하려면
  아래 패키지가 필요합니다.

  pip install xhtml2pdf pymupdf numpy

  - xhtml2pdf: HTML/CSS → PDF 변환 (순수 Python, 외부 바이너리 불필요). markdown
  패키지로 .md를 HTML로 바꾼 뒤 이걸로 PDF 생성.
  - pymupdf (import명 fitz): PDF → PNG 래스터화. poppler 설치 없이 DPI 지정해서
  바로 이미지로 뽑을 수 있음.
  - numpy: grain(노이즈) 합성용. paddlepaddle 의존성으로 이미 깔려 있을 가능성
  높지만, 명시적으로 넣어도 무방.

  markdown, pillow는 이미 설치돼 있어 추가 설치 불필요합니다.

  참고로 한글 폰트는 pip 패키지가 아니라 시스템 폰트 파일(맑은고딕 등) 경로를
  직접 xhtml2pdf에 등록해야 해서, 구현 시 C:\Windows\Fonts 폴더에서 자동
  탐색하도록 코드에 넣을게요.

  이대로 설치 진행하고 코드 작성 시작해도 될까요?
